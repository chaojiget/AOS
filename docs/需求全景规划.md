# Agent OS 需求全景规划（最终基线）

> 目标：汇总 `docs/` 与 `docs/需求草稿/` 中的既有内容，沉淀为「项目最终形态」的需求蓝图，供产品、技术与运营在实现、评审与运维阶段共用。
> 范围：覆盖 v0 最小核、阶段一（日志闭环）与阶段二（技能固化）三大阶段，并给出功能拆解、非功能约束、数据/接口契约与验收口径。

---

## 1. 背景与愿景

- **愿景**：打造一个可自感知、自反思、自改进的 Agent Operating System，在本地环境即可跑通从实时执行到离线回放、从日志沉淀到技能固化的完整闭环。
- **驱动因素**：
  - 团队需要一套标准化的日志契约与运行机制，降低 Agent 行为不可控的风险；
  - 通过 Episode 与 Replay 建立“可回放”的质量基线，支撑迭代评估与问题追溯；
  - 将高频工具调用提炼为技能模板，提高重复任务的成功率与成本效率。
- **关键 Stakeholder**：PO（价值与优先级）、TL（架构与守护）、体验负责人（聊天与日志体验）、技能运营者（技能审核与上线）、Guardian（预算/SLA 与安全审计）。

## 2. 阶段化里程碑

| 阶段 | 时间盒 | 核心目标 | 关键产物 | 验收事件 |
| ---- | ------ | -------- | -------- | -------- |
| v0 最小核 | 2 周 | 建立 `perceive → plan → act → review → final` Loop，跑通 Episode 追加写与回放 | `core/agent.ts`、`runtime/episode.ts`、CLI `run/replay` | `pnpm run replay <trace_id>` 可复现示例 CSV→统计→Markdown 任务 |
| 阶段一：日志闭环 | 2 周 | NestJS API + SSE/WS 实时日志；Next.js 聊天/LogFlow；SQLite 持久化 runs/events | `servers/api`、`servers/replay`、`ui/chat`、`ui/logs`、Drizzle schema | 聊天页实时展示 plan/tool/reflect 事件，Episode 离线回放一致 |
| 阶段二：技能固化 | 2 周 | 日志分析→技能模板→人工审核→灰度上线；指标面板与 Guardian 守护 | `packages/skills/pipeline.ts`、`ui/skills`、`scripts/skills-eval.ts`、指标面板 | 触发 `POST /api/skills/analyze` 产出 ≥1 个审核通过技能并被后续 run 调用 |

> 每个阶段均需保留 Episode/Git 证据，并通过 `pnpm typecheck && pnpm test` 作为发布前置门禁。

## 3. 用户场景与 Jobs-to-be-done

1. **Agent 使用者（操作员）**：在聊天界面提交任务，实时观察计划、工具调用、反思与最终答案，必要时审批敏感操作。
2. **技能运营者**：对高频日志片段进行聚合、模板化与审核，控制技能启用节奏并监控其收益/成本。
3. **Guardian / 管控者**：设定预算、步数、风险阈值；监控运行异常、工具失败与安全告警；决定是否暂停或回滚。
4. **工程团队（FE/BE/TL）**：实现 API/Loop/UI，保障日志契约与性能指标；维护测试、回放、指标看板。

## 4. 功能需求全景

### 4.1 RunLoop 与执行闭环
- 维持 `Observe/Think/Act/Reflect/Stop` 状态机，支持工具失败、无计划兜底、技能短路等分支。
- 事件发射：`run.*`、`plan.updated`、`tool.started/finished/failed`、`reflect.note`、`skill.used`、`guardian.alert`、`final.answer`。
- 支持 Guardian 注入预算/步数限制，命中阈值时记录 `run.terminated` 并保留上下文。
- Episode Logger 采用 JSONL 追加写，含 `ts/type/payload/cost/duration/token` 字段；回放需保证事件顺序与实时一致。

### 4.2 API 与前端体验
- **API**：
  - `POST /api/agent/start` 启动任务，返回 `{ runId }`；
  - `GET /api/runs/:runId` 读取摘要；
  - `GET /api/runs/:runId/stream` SSE 实时事件（支持心跳、防断线重连）；
  - `GET /api/runs/:runId/events?since=ts&limit=100` 历史分页；
  - `POST /api/skills/analyze`、`GET /api/skills`、`POST /api/skills/:id/enable`；
  - `POST /api/mcp/register`、`DELETE /api/mcp/:id` 管理工具配置。
- **前端**（Next.js 单页 + 多页扩展）：
  - 聊天页：输入区、计划/行动/反思流、技能面板、预算/步数状态；
  - 日志页：实时流 + 历史分页、事件过滤（按类型/关键字）、Episode 下载；
  - 技能页：候选列表、模板详情、审核操作、胜率/使用频次图表；
  - Guardian 面板：预算阈值设置、告警记录、回放入口。
- 无障碍：键盘可达、语义化标签、颜色对比符合 WCAG AA；性能目标 TTI ≤ 3s、SSE 延迟 < 1s。

### 4.3 技能固化流水线
- 日志聚合：按工具调用归档样本，生成 `call_id` 去重列表，统计成功率、平均耗时、token 成本。
- 模板生成：LLM 将样本转成 `template_json`（含参数 schema、输入输出示例、触发条件、风险提示）。
- 审核工作流：状态机 `draft → pending_review → approved/rejected`，需记录审核人与备注；可灰度启用（配置流量百分比）。
- 上线监控：统计 `used_count`、`win_rate`、`avg_cost`，低于阈值触发自动下线与告警。
- 离线评估：`scripts/skills-eval.ts` 读取 Episode 片段进行回放评估，输出报告至 `reports/skills/*.md`。

### 4.4 可观测性与守护
- Pino 结构化日志 + 指标表（成功率、步数、工具命中、token、成本、延迟）。
- OpenTelemetry 追踪（可选）：RunLoop 每步生成 span，前端可展示时序图。
- 告警：Guardian 将严重事件（预算超限、敏感操作、技能回退）推送至 Slack/Webhook。
- 数据留存：Episode/事件保留 30 天，支持脱敏钩子；敏感数据标记并限制导出。

## 5. 非功能需求

| 类别 | 约束 |
| ---- | ---- |
| 性能 | 单任务总时延 p95 ≤ 60s；SSE 首帧 ≤ 1s；数据库查询 p95 ≤ 200ms |
| 可用性 | API 与 SSE 具备自动重连/降级；工具调用超时重试 2 次 |
| 安全 | 工具白名单 + 参数校验；敏感操作触发 `user.confirm.request`；Secrets 仅存 `.env` |
| 可维护性 | 所有契约以 TypeScript 类型导出；CLI/脚本提供自检；运行脚本要求 `pnpm typecheck/test/replay/smoke` 通过 |
| 可扩展性 | MCP 工具接口模块化；技能模板存储兼容 SQLite/MySQL；RunLoop 支持插拔策略 |

## 6. 数据模型与契约

- **数据库（Drizzle + SQLite）**：`runs`、`events`、`skills`、`skill_versions`、`skill_runs`、`memories`、`mcp_configs`、`evals`、`ab_tests`。
- **事件结构**：
  ```json
  {
    "id": "ulid",
    "run_id": "ulid",
    "ts": "RFC3339",
    "type": "plan.updated|reflect.note|tool.started|tool.succeeded|tool.failed|skill.used|guardian.alert|run.finished",
    "status": "info|success|error|warning",
    "title": "string",
    "payload": { "free": "object" },
    "token_input": 0,
    "token_output": 0,
    "cost_usd": 0.0,
    "duration_ms": 0
  }
  ```
- **技能模板**：
  ```json
  {
    "id": "web.extract.v1",
    "name": "WebExtractSkill",
    "version": 1,
    "trigger": { "tool": "mcp.playwright.goto", "match": "url.includes('news')" },
    "args_schema": { "$schema": "https://json-schema.org/draft/2020-12/schema", "type": "object", "properties": {"url": {"type": "string"}} },
    "sample_inputs": [ { "url": "https://example.com" } ],
    "sample_outputs": [ { "markdown": "..." } ],
    "guardrails": [ "仅读取公开页面", "禁止写操作" ],
    "review_status": "pending_review",
    "metrics": { "win_rate": 0.82, "avg_cost": 0.12, "avg_latency_ms": 1420 }
  }
  ```

## 7. 验收标准（DoD）

1. **A1 - 实时执行可视化**：
   - Given：用户在前端聊天页提交任务；
   - When：RunLoop 执行并产出 plan/tool/reflect/final 事件；
   - Then：前端实时显示事件流，最终展示 `FINAL_ANSWER`，Episode 落盘且含完整事件。
2. **A2 - 离线回放一致**：
   - Given：存在一条完整 Episode；
   - When：执行 `pnpm run replay <runId>`；
   - Then：回放输出与实时事件序列一致，并生成对比报告。
3. **A3 - 技能固化闭环**：
   - Given：最近 N 次任务包含重复工具调用；
   - When：触发技能分析并通过审核启用；
   - Then：后续任务自动调用该技能，指标面板展示使用/胜率数据。
4. **A4 - 守护与告警**：
   - Given：Guardian 设置预算/步数阈值；
   - When：任务超过阈值或触发敏感操作；
   - Then：系统记录告警、必要时暂停执行，并在 UI 与 Webhook 中可见。

## 8. 指标与治理

- **北极星指标**：任务成功率 ≥ 70%，并具备离线回放证据。
- **护栏指标**：成本 ≤ 1.5 USD / 任务；步数 p95 ≤ 12；技能 win_rate ≥ 70%；敏感操作人工审批响应 ≤ 2 分钟。
- **监控面板**：成功率、平均步数、工具命中率、技能收益、SSE 延迟、Episode 写入耗时。
- **治理流程**：Intake → Discovery → Plan → Implement → Review → Patch-Once → Stabilize → Release → Observe → Learn（详见 `docs/PROJECT.md`）。

## 9. 风险与缓解

| ID | 风险 | 影响 | 缓解策略 |
| -- | ---- | ---- | -------- |
| R1 | 日志/回放不一致 | 影响审计与回归 | 引入 Golden Logs 基线，`pnpm replay` 纳入 CI，关键事件编写单测 |
| R2 | 技能误判导致退化 | 任务失败率上升 | 滑窗评估 + 自动下线；人工审核记录；灰度限制流量 |
| R3 | 工具调用超时或故障 | 循环停滞 | 超时/重试机制；降级路径（无计划兜底）；Guardian 审批敏感重试 |
| R4 | SSE/WS 在生产受限 | 无法实时展示 | 预留轮询降级；部署前压测，必要时启用 WebSocket 兼容 |
| R5 | 数据敏感/泄漏风险 | 合规违规 | 数据脱敏钩子；Episode 导出权限控制；Secrets 管理标准化 |

## 10. 发布策略与运维

- **推广流程**：本地冒烟 → 开发环境 5% 金丝雀（≥100 次任务）→ 全量上线。
- **回滚准则**：成功率或成本指标退化 > 2×；技能 win_rate < 50% 且不可快速修复。
- **运维脚本**：`pnpm intake`（需求校验）、`pnpm typecheck`、`pnpm lint`、`pnpm test`、`pnpm smoke`、`pnpm replay`、`pnpm skills:eval`。
- **知识沉淀**：Episode 与技能评估报告入 `reports/`; 经验更新 `AGENTS.md` 与 `docs/需求全景规划.md`。
